# nvocc-mailer 踩坑记录

## 1. Docker 相关

### MySQL 认证插件问题
**现象**：pymysql 连接 MySQL 8.0 报错 `caching_sha2_password` 认证失败。  
**原因**：MySQL 8.0 默认使用 `caching_sha2_password`，pymysql 不支持，需要安装 `cryptography` 包或改用旧插件。  
**解决**：MySQL 容器启动命令加 `--default-authentication-plugin=mysql_native_password`，同时必须 `docker compose down -v` 删掉旧 volume 重新初始化，否则旧认证方式残留。

### 时区问题
**现象**：数据库里存的时间比北京时间慢 8 小时。  
**原因**：Docker 容器默认 UTC，MySQL `NOW()` 用容器时区。  
**解决**：app 容器加环境变量 `TZ=Asia/Shanghai`，同时 pymysql 连接加 `init_command="SET time_zone='+08:00'"`，两个缺一不可。

### docker-compose.yml 不在服务器上
**现象**：服务器上找不到 docker-compose.yml，无法用 `docker compose up`。  
**原因**：当时直接手动 `docker run` 启动的两个容器，没有把 compose 文件传上去。  
**教训**：每次重建容器要手动跑完整的 `docker run` 命令，命令模板保存在部署文档里。

### docker cp 更新代码后重建容器会回退
**现象**：用 `docker cp` 更新了 app.py，重建容器后代码回退到旧版本。  
**原因**：`docker cp` 只改了运行中容器的文件，没有改镜像。重新 `docker run` 时用的是旧镜像。  
**解决**：改完代码必须重新 build 镜像才能固化；临时修改用 `docker cp`，正式发布必须打包镜像。

### build context 过大导致卡死
**现象**：`docker build` 一动不动，Ctrl+C 后 Docker Desktop 崩溃。  
**原因**：项目目录里有 629MB 的 `nvocc-mailer.tar.gz`，Docker 把它全部打进 build context 发给引擎。  
**解决**：根目录加 `.dockerignore`，排除 `*.tar.gz`、`uploads/` 等大文件。  
**注意**：`docker build` 卡住不要 Ctrl+C，会误杀 Docker 引擎进程导致崩溃。

## 2. SMTP / 邮件发送

### 端口 5000 被占用
**现象**：容器启动后访问 5000 端口报错。  
**原因**：服务器上另一个服务（vessel_http_api）占用了 5000 端口。  
**解决**：docker-compose.yml 改为 `5001:5000`。

### 企业邮件服务器 STARTTLS
**现象**：`SMTP_USE_SSL=false` 时代码会尝试 STARTTLS，如果服务器不支持会导致全部失败。  
**当前状态**：192.168.32.8:25 支持 STARTTLS，实测可用，未修改。  
**潜在风险**：换邮件服务器时需确认是否支持 STARTTLS，否则需要改 connect() 函数。

## 3. 退信检测

### IMAP → POP3
**现象**：企业邮件服务器不支持 IMAP，只开了 POP3。  
**解决**：把 `imaplib` 全部替换为 `poplib`，接口不同但逻辑类似。

### POP3 端口和 SSL
**现象**：最初用 995（SSL）连接失败。  
**原因**：服务器 POP3 走 110 端口不加密，不是 995。  
**解决**：`POP3_PORT=110`，`POP3_USE_SSL=false`。

### HTML 标签把退信邮箱地址一起删掉
**现象**：退信检测始终返回 0，但收件箱里明明有退信。  
**原因**：企业邮件服务器退信格式是 `<testb@example.com>:` 这种尖括号格式，代码先做 HTML 标签剥离（`re.sub(r'<[^>]+>', ' ', body)`），把邮箱地址一起删了，导致正则匹配不到。  
**解决**：把英文格式的正则匹配移到 HTML 剥离之前执行。

### POP3 loop 里用 break 可能漏退信
**现象**：POP3 遍历时遇到早于批次时间的邮件就 break 停止，如果服务器返回顺序不是严格时间序，会漏掉退信。  
**解决**：改为 `continue` 跳过，遍历全部邮件，稍慢但不会漏。

### 退信检测多进程失效（已修复）
**现象**：最初用内存字典 `_pause_flags` 存暂停标志，Gunicorn 多 worker 时，接收暂停请求的 worker 和发邮件的 worker 内存不共享，暂停无效。  
**解决**：改为数据库存状态，发邮件线程每封发完后查一次数据库的 status 字段。

## 4. 时区与时间比较

### 退信时间对比 bug
**现象**：退信检测跑了但检测不到，加日志发现所有退信邮件都被过滤掉了。  
**原因一**：`mail_time.replace(tzinfo=None)` 去掉时区后直接比较，时区信息丢失导致错误判断。  
**原因二**：`batch_start.replace(tzinfo=timezone.utc)` 把本地时间（UTC+8）当成 UTC 处理，偏差 8 小时，所有退信被判断为"太早"。  
**解决**：用 `.timestamp()` 比较，Python 对 naive datetime 默认用本地时区，对 aware datetime 转成 UTC，两者用同一基准比较，结果正确。

## 5. 前端

### Quill.js 字号 HTML 标签冲突
**现象**：发送 HTML 邮件时，退信检测的 HTML 剥离把 Quill 生成的 `<span style="font-size:...">` 也剥掉了，正文显示异常。  
**说明**：退信检测只对退信邮件原文做 HTML 剥离，不影响发件正文，两条代码路径分开，实际无冲突。

## 6. 数据库迁移

### 新增字段兼容旧数据
**场景**：上线后新增 `extra_data` 字段，已有的历史数据没有这个字段（NULL）。  
**解决**：启动时自动检测字段是否存在，不存在则 `ALTER TABLE` 补上；下载 CSV 时对 NULL 的旧数据走兜底逻辑（只输出公司名/邮箱/退信原因三列）。

### migrate_db 启动时 DB 未就绪
**现象**：容器启动时 MySQL 还没完全初始化，`migrate_db()` 报错导致 Gunicorn worker 崩溃重启。  
**解决**：用 `try/except` 包住启动时的迁移调用，第一次失败没关系，后续请求触发时 DB 已就绪。
